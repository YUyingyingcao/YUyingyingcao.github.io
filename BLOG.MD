```md
我想从 0 开始做一个个人博客，请你直接协助我规划、实现、部署。

## 基本目标

我要给朋友做一个类似 Yuimi Lab 气质的个人博客，但不是复制 Yuimi Lab，而是根据朋友的人设、爱好、头像、图片素材重新设计。

目标风格：
- 二次元
- 清新
- 可爱
- 日杂 / 手账 / 个人空间感
- 有堆料感
- 有互动
- 有新鲜感
- 有创意设计
- 能写文章、展示项目、展示兴趣内容
- 适合静态部署到 GitHub Pages 或类似平台

不要做成普通模板站、简历站、纯技术站、黑色极客站、赛博终端站。

## 技术方向

优先使用：
- Astro
- 静态博客
- Markdown / MDX 写文章
- GitHub Pages 部署
- GitHub Actions 自动构建

如果有更合适的方案，可以提出，但请优先考虑轻量、稳定、静态、可长期维护。

从 0 开始时，请你先帮我：
1. 确认项目目录和初始化方式。
2. 搭建 Astro 项目。
3. 做基础页面：
   - 首页
   - Blog
   - Projects / Works
   - About
   - Games 或 Playground
   - 404
4. 配好 Markdown 文章系统。
5. 配好部署到 GitHub Pages。
6. 每次改完都运行构建验证。

如果安装依赖，请使用国内镜像，例如：
```bash
npm install xxx --registry=https://registry.npmmirror.com
```

## 视觉设计总原则

整体要像一个“可以被慢慢翻看的二次元个人空间”，不是一个产品官网。

关键词：
- 奶油白
- 浅蓝
- 淡粉
- 浅藤紫
- 柔和墨蓝
- 半透明
- 玻璃边缘
- 贴纸
- 便签
- 胶带
- 小装饰物
- 手账排版
- 清透空气感

避免：
- 大片黑色卡片
- 霓虹赛博风
- 终端极客风过重
- 大量纯白硬块
- 一整页都是同一种玻璃框
- 装饰物重复滥用
- 太像 AI 写给用户的项目汇报文案
- 英雄区做成普通营销落地页
- 过大的“介绍功能”文字
- 让页面变成模板感很强的作品集

## 首页设计要求

首页要是可玩的、好看的、有记忆点的，不要只是文章列表。

可以包含：
- 沉浸式首屏大图
- 头像 / 个人卡片交互
- 终端输入效果，但不能太黑、太硬核
- 鼠标交互
- 背景动画
- 小彩蛋
- 可爱的导航
- 个人身份展示
- GitHub / Bilibili / Steam / X / 其他社交按钮
- 音乐播放器
- 看板娘 / Live2D
- 小游戏入口
- 右键菜单
- 自定义滚动条
- 页脚计时器或纪念日计时

首页头图下方区域不要只是大块重复卡片，要规划成“展示 + 娱乐 + 个人气质”的连续空间。

可玩入口不要和导航重复。导航负责去页面，可玩入口应该更像彩蛋、展柜、工具台、房间摆件。

## 导航设计要求

导航可以做成胶囊、抽屉、悬浮条、贴纸条等形式。

要求：
- 默认状态可以收起
- 鼠标靠近展开
- 每个分类有图标
- hover 有反馈
- 图标可以有小动画
- 不要遮挡文字
- 不要使用会压住内容的终端三圆点装饰

建议分类：
- HOME
- BLOG
- GAME / PLAY
- WORKS / PROJECTS
- ABOUT / ME
- 可按朋友特点增加 MUSIC、GALLERY、DIARY 等

## Markdown / 文章页要求

文章页要适合长期写教程和随笔。

请默认安装或配置：
- Markdown 美化
- GFM 支持
- 代码块美化
- 复制按钮
- 标题自动锚点
- 文章内目录
- 文章搜索
- sitemap

推荐：
- `astro-expressive-code`
- `remark-gfm`
- `rehype-slug`
- `rehype-autolink-headings`
- `pagefind`
- `@astrojs/sitemap`
- `astro-icon`

文章页视觉：
- 像纸张阅读页
- 清爽
- 字体可读
- 代码块可见度高
- 长文章要有右侧目录
- 移动端不能溢出
- 表格、引用、列表都要有样式

## 图片和素材规则

如果我提供：
- 头像
- 背景图
- 文章封面
- icon
- loading 图
- 404 图
- 旧博客资源

请你整理到合适的 `public/assets/` 目录中。

文章封面规则：
- 如果文章数量不超过封面图数量，不允许复用封面。
- 随机分配可以，但不能重复。
- 裁切前要确保人物脸部露出来。
- 图片要尽量优化成 webp。
- 不要随便外链图床，优先本地静态资源。
- 如果用外部图床，要先验证图片能访问，不要出现空白和 404。

## 交互功能可选清单

可以逐步实现，不要求一次全部做完：

### 基础交互

### 个人展示

- 随机一句话

### 娱乐
- 音乐播放器
- 旋转唱片
- 播放进度缓存
- Live2D 看板娘

- 隐藏彩蛋

### 内容
- Blog 时间线

## 动效规则

动效要轻巧，有手感，但不能影响阅读。

喜欢：
- 弹簧感
- 浮力感
- 下拉刷新那种释放感
- 微吸附
- 小物件轻微摆动
- hover 反馈明确

避免：
- 一帧一帧卡顿感
- 滚很久才触发
- 强行锁滚动
- 页面滚动和元素动画互相打架
- 动画导致布局错位
- 动画遮挡正文

如果实现复杂滚动动画，请优先保证：
1. 不锁死滚轮。
2. 不导致页面无法继续滚动。
3. 上下滚动时状态一致。
4. 移动端不要炸布局。

## 文案规则

文案要像朋友自己的个人空间，不要像开发报告。

避免：
- “我为你构建了……”
- “这里展示你的功能……”
- “这是一个现代化博客……”
- “用于体现技术属性……”

更希望：
- 像手账标题
- 像房间分区
- 像兴趣收藏
- 像个人留言
- 简短、有气质、有一点点可爱

## 代码和工程规则

请你：
- 先读项目结构，再改。
- 用现有风格，不要乱引入大型框架。
- 需要编辑文件时直接实现，不要只停留在建议。
- 每次重要改动后运行：
  ```bash
  npm run build
  ```
- 如果需要本地预览，启动：
  ```bash
  npm run dev -- --host 127.0.0.1
  ```
- 不要使用破坏性 Git 命令。
- 不要重置我的改动。
- 如果工作区有未提交文件，先说明并谨慎处理。
- PowerShell 下不要使用 `&&`，分开执行命令。

## 部署规则

目标优先：
- GitHub Pages
- GitHub Actions 自动部署

部署时请：
1. 生成正确的 Actions workflow。
2. 确认不是让 GitHub Pages 默认 Jekyll 构建源码。
3. 避免出现 `.astro` 被 Jekyll 当 YAML 解析的错误。
4. 构建成功后再 push。
5. 如果图片在本地能显示但线上不显示，检查路径大小写、public 目录、base 路径、Git 是否提交。
